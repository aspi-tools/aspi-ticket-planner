# No31 手続き最適化アプリ — 設計プラン v0.1

## Context（なぜ作るか）

ASPIRESTのチケット制度は **回数変更がいつでも可能 + チケットは繰り越し蓄積** という設計のため、ダウンセル/退会の場面で「残チケットをどう消化するか」がトーク上の論点になる。

スタッフが対面提案中に **「このペースで通えば◯月までに消化できる」「今やめたら◯円分損する」** を即座に視覚提示できれば、

- ダウンセル抑制（→回帰提案へ誘導）
- 退会阻止（→残チケットの金額インパクト）
- 将来アップセル（消化計画から逆算した頻度提案）

につながる。

既に Google Sheets 上に「退会＆ダウンセル阻止ツール」(`1zbsuohuo3aW5z6g7iSBx1A61z9qpXAo6RDC0Uf0sZPM`) が存在し、計算ロジックは確立済み。ただし **スプシUIは顧客に見せる前提で作られておらず、iPad対面で使うには操作性・視覚インパクトが弱い**。

→ 既存の計算ロジックを踏襲しつつ、**iPad対面提案に最適化したWebアプリ** として作り直す。No01カロリー計算機（GitHub Pages公開）と同じ配信モデル。

## 既存資産の分析（再利用するもの）

### スプシ「退会＆ダウンセル阻止ツール」の計算ロジック

**1. 残チケット金額計算（損失可視化）**
- 入力: 残チケット（分）
- 単価マトリクス（4区分）:
  | 区分 | 単価(¥/分) |
  |---|---|
  | 通常会費（月8以上） | 117 |
  | 通常会費（月8以下） | 127 |
  | 旧会費（月8以上） | 106.5 |
  | 旧会費（月8以下） | 108 |
- 出力: 金額 = `FLOOR(残分 × 単価, 100)`
- 注記: 「会費改定未対応」「若干ズレますがほぼ同値」 → アプリ版で会費改定後単価を反映する

**2. ダウンセル回帰シミュレーター**
- プラン↔分換算（マスタ「触らない」シート）: 月N → N×30分
- 入力: 残チケット（分）／手続き後月回数（月X）／消化予定（月あたり通う分）／手続き適用月
- 出力: 消化しきるまでの月数 / 回帰手続き締切日（適用月+消化月-1の10日）
- ロジック: `消化月数 = CEILING(残 / (消化予定 - 月X付与分))`

これらの**計算式はそのまま流用**する。会費単価マスタは要更新（要：最新会費の確認）。

### 既存類似アプリの配信パターン（No01カロリー計算機）
- リポジトリ: `aspi-tools/aspi-calorie-calc`
- 公開URL: `https://aspi-tools.github.io/aspi-calorie-calc/`
- QRコード経由でiPadで開く運用
- HTML/CSS/JS のみ（ビルド不要）
- → **同じ運用モデル**を採用する（リポジトリ名候補: `aspi-procedure-optimizer` / `aspi-ticket-tool`）

### 制約（再利用しないもの・避けるもの）
- /tetsuzuki-log スキル：手続き発生**後**のKPI記録用なので**スコープ外**（このアプリは手続き発生**前**の提案ツール）
- 退会KPI管理.xlsx 連携：MVP不要
- 顧客データの自動取込（ASPI Smile/デジム）：MVP不要、当面は手入力

## 設計

### 技術スタック

| 層 | 採用 | 理由 |
|---|---|---|
| 配信 | GitHub Pages (`aspi-tools` org) | No01踏襲。iPad Safariで開けば即動く。サーバー不要 |
| 構成 | 静的HTML/CSS/JS（ビルドなし） | iPadで `https://...` を開くだけ。メンテ容易 |
| UI | Tailwind CSS (CDN) + 手書きJS | 速い・H01でも実績 |
| グラフ | Chart.js (CDN) | 残チケット消化タイムライン用。軽量 |
| 状態保持 | なし（ステートレス） / `localStorage` で直近入力のみ | 顧客データを残さない設計（個人情報リスク回避） |

### 機能（2モード切替UI）

**Mode A: 消化シミュレーター（ダウンセル提案 / アップセル提案）**

入力エリア（大きいタップ領域・iPad最適化）:
- 現プラン（月N／プルダウン: 月1〜月24）
- 残チケット（分単位 or 回数単位、トグル切替）
- ダウンセル後プラン候補（月N、複数選択可：最大3つ比較）
- 月次通う頻度（消化分／月）

出力エリア:
- **タイムラインチャート**（横軸: 月、縦軸: 残チケット分）
  - 各候補プランで「いつ完全消化するか」を線で表示
  - 完了月の縦線＋ラベル
- 各候補ごとのサマリカード:
  - 「このプランなら **◯ヶ月** で消化完了」
  - 「最終手続き締切: **YYYY/MM/DD**」
  - 「月平均 ◯分（◯回）通えばOK」
- ポジティブ提案文（コピペ可）テンプレ:
  > 「現状の残チケットは◯分。月◯ペースで通えば◯ヶ月で消化できます。今のうちに月◯プランへ調整しておくと、計画的に使い切れますよ」

**Mode B: 損失可視化（退会阻止）**

入力エリア:
- 現プラン区分（4区分プルダウン: 通常×月8以上/以下、旧×月8以上/以下）
- 残チケット（分単位）

出力エリア（インパクト重視デザイン）:
- 画面中央に **巨大金額** 表示（赤系/グラデ）
  > **¥21,000** 相当のチケットが残っています
- 換算情報:
  - 「◯時間（◯回）分」
  - 「◯回分のセッションがまだ受けられます」
- 比喩トーク用テンプレ（コピペ可）:
  > 「今退会されると、残ってる◯分（¥◯）のチケットが消滅します。月◯ペースなら◯ヶ月でしっかり使い切れますが、いかがでしょうか？」
- 「Mode Aで消化計画を見る」ボタン → Mode Aへ遷移＆値引継ぎ

### 共通仕様

- iPad（11インチ想定）レイアウト最優先：縦/横どちらでも崩れない
- 顧客名/会員番号入力欄（任意・印刷時のみ使用）
- 「リセット」ボタン
- ヘッダーにモード切替タブ（消化シミュ / 損失可視化）
- 単価マスタは `data/pricing.json` で管理し、会費改定時に1ファイル更新で済む構造

## 重要な確認事項（実装前に潰す）

1. **会費単価マスタの最新版** — 既存スプシは「会費改定未対応」と注記あり。改定後の単価表が必要。
2. **「分単位 vs 回数単位」入力の好み** — スタッフは普段どちらで会話しているか（30分=1回換算は内部既知だが、UIをどちらに寄せるか）
3. **公開先リポジトリ** — `aspi-tools` orgに新規リポジトリを切る前提でOKか
4. **会員番号入力の要否** — 個人情報をブラウザに残したくない場合は印刷時のみテキスト入力にとどめる

これらは Phase 0.2 直前に詰める（実装着手の前段で確認）。

## 検証

MVP完成時の検証手順:

1. **ロジック検証**: 既存スプシ「退会＆ダウンセル阻止ツール」と同じ入力で同じ出力になるかをテーブルで突合（残180分・通常月8以上 → ¥21,000 / 残1020分・月4ダウン・月360消化 → 5ヶ月・2026/07/10 等）
2. **iPad実機検証**: 神楽坂iPadでSafariから開く→入力→表示までスムーズか（タップ領域・キーボード・横画面）
3. **対面リハ**: スタッフ1名と模擬ロープレ→「顧客視点で金額インパクトが伝わるか」「消化計画グラフが直感的か」フィードバック取得
4. **トーク文テンプレ**: コピペでLINE/Slackに貼り付けられる体裁か確認
